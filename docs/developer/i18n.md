# Internationalisation

Granite externalises every user-visible string through a small i18n module
at `src/core/i18n/`. Adding a locale is a one-file change; adding a string
is a two-line change (one in the source-of-truth English table, one at the
call site).

## Architecture

```text
src/core/i18n/
├── index.ts                  Public API: t, registerLocale, setLocale, …
├── direction.ts              localeDirection + per-note dir frontmatter
├── date-format.ts            Locale-aware date / time formatting
├── externalization.test.ts   No-hard-coded-strings audit
├── locale-coverage.test.ts   Locale completeness audit
├── locales/
│   ├── index.ts              BUILTIN_LOCALES map
│   ├── en.ts                 EN_LOCALE — the source of truth
│   └── he.ts                 HE_LOCALE — RTL locale, 100% coverage today
└── index.test.ts             Unit tests for t(), plural, pseudo
```

## The locale registry

`registerLocale(id, strings)` adds or replaces a locale map. The built-ins
are registered at module init from `locales/index.ts`:

```ts
import { registerLocale, type LocaleMap } from "@core/i18n";
registerLocale("nl", { "command.palette.title": "Opdrachtenpalet", /* … */ });
```

A locale is a flat `Record<string, string>` keyed by dotted identifier. The
English table (`EN_LOCALE` in `locales/en.ts`) is the source of truth: any
key missing from another locale falls back to the English value, and
`verifyLocaleSync()` reports missing/extra keys per locale (used by
`locale-coverage.test.ts`).

The active locale is persisted in `localStorage` under
`granite.locale.v1`. `setLocale(id)` switches and emits to subscribers.
`getLocale()` returns the current id; `listLocales()` returns every
registered id.

## `t(key, params?)`

```ts
import { t } from "@core/i18n";

t("command.palette.title");
// "Command palette"

t("editor.savedAt", { time: "12:34" });
// "Saved at 12:34"

t("vault.fileCount", { count: 1 });
// "1 file"
t("vault.fileCount", { count: 7 });
// "7 files"
```

- `{name}` placeholders are replaced after lookup.
- ICU-style plural blocks (`{count, plural, one {1 file} other {{count} files}}`)
  are resolved with `count`-based selection — `one` for 1, `other`
  otherwise. The implementation is intentionally small (no full ICU MF).
- Missing keys fall through to English, then to the literal key.

For migration logic that needs the source English text directly (e.g. a
plugin that wants to provide its own translations on top), use
`getDefaultLocaleText(key)`.

## Locale direction

`src/core/i18n/direction.ts`:

```ts
import { localeDirection, noteDirectionFromFrontmatter } from "@core/i18n/direction";

localeDirection("ar");        // "rtl"
localeDirection("en-US");     // "ltr"
noteDirectionFromFrontmatter({ dir: "rtl" });  // "rtl"
```

`LocaleDirectionBinder` (`src/ui/LocaleDirectionBinder.tsx`) subscribes to
`subscribeI18n` and sets `dir` / `lang` on `<html>` accordingly. Per-note
overrides come from the note's `dir` frontmatter and are applied to the
editor and reading-view roots.

Today RTL locales include `ar`, `fa`, `he`, `ur`. Adding a new RTL locale
means adding it to `RTL_LOCALES` in `direction.ts` and shipping its
locale table.

## Date formatting

`src/core/i18n/date-format.ts` wraps `Intl.DateTimeFormat` with helpers
that honor the active locale and the user's date-format preference (set in
Settings → Files and links → Date format). Always go through these helpers
when displaying a date; never hand-roll `toLocaleString` calls in UI code.

## Adding a string

1. Add the key + value to `src/core/i18n/locales/en.ts`.
2. Call `t("…")` at the use site.
3. (Optional) Add translations to other locale files. `verifyLocaleSync()`
   surfaces the gap if you skip this.

If you do not externalise a string, the externalization test will catch it.

## Externalization test

`src/core/i18n/externalization.test.ts` scans user-visible UI code and fails
if it finds a hard-coded English string. The test has a small allowlist for
brand names and code tokens (see `PSEUDO_VALUE_ALLOWLIST` in
`src/core/i18n/index.ts`). When the test fails:

- If the string is user-visible, add it to `en.ts` and use `t(...)`.
- If the string is not user-visible (e.g. a `data-*` attribute, a console
  message), refactor it out of the JSX/text content or extend the allowlist
  in the test with a comment explaining why.

## The pseudo locale

`PSEUDO_LOCALE = "__pseudo__"` is a synthetic locale that wraps every
English value in `[ … ]` and substitutes accented characters for vowels and
a couple of consonants. Switch to it from Settings → Appearance → Locale.
Anything that bypassed `t(...)` appears without brackets, immediately
sticking out at runtime.

The transform skips code-like values (single tokens containing digits, `+`,
`-`, `_`, `.`, `:`, `/`) and an explicit allowlist of brand and format
strings: `Granite`, `Obsidian`, `Markdown`, `Ctrl+P`, `YYYY-MM-DD`,
`HH:mm`, `attachments`. Even allowlisted values get bracketed so they are
still distinguishable from strings that skipped `t(...)` entirely.

`{placeholder}` segments are preserved verbatim — pseudo-translated
strings still interpolate correctly.

## Coverage and audits

- `locale-coverage.test.ts` — enforces that every non-pseudo locale either
  ships full coverage or stays gracefully partial; reports missing and
  extra keys via `verifyLocaleSync()`.
- `externalization.test.ts` — catches hard-coded user-visible strings.
- `verify:i18n-browser` — exercises locale switching, RTL flip, and the
  pseudo locale in a real browser.

## Adding a new locale

1. Create `src/core/i18n/locales/<id>.ts` exporting a `LocaleMap`.
2. Register it in `src/core/i18n/locales/index.ts` so it lands in
   `BUILTIN_LOCALES`.
3. If the locale is RTL, add it to `RTL_LOCALES` in `direction.ts`.
4. Run `bun run test src/core/i18n` — `locale-coverage.test.ts` will
   report missing keys against `EN_LOCALE`.
5. Run `verify:i18n-browser` to confirm runtime switching works.

The 1-file convention exists deliberately — translations are reviewable in
a single PR, and a partial locale ships safely thanks to the English
fallback.
